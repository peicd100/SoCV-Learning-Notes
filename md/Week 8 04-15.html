    07. Advanced SAT Techniques

    Recap of the Modern SAT Algorithm

    我現在回頭看 modern SAT algorithm 的那 13 行 pseudo code，已經不會把它當成一段單純的流程圖了。它其實把整顆 CDCL solver 的核心節奏都寫出來了：先看當前 target 在 BCP 之下會不會立刻矛盾，如果沒有，就進入決策迴圈；每做一個 decision，就做一次 propagation；只要 propagation 撞到 conflict，就進入 conflict analysis、學一條 clause、然後 backtrack 到更有意義的位置。

    我很喜歡這個版本的寫法，因為它非常誠實地告訴我 SAT solver 不是一個神奇黑盒，而是一個反覆在做「猜測、推出、撞牆、學習、重試」的程序。真正讓它厲害的，不是它從不犯錯，而是每次犯錯都會把失敗經驗留下來，讓下一次搜尋不要再白走。

    另一個我很有感的地方，是 solver 的終止條件其實很乾淨。若在 decision level 0 就發生 conflict，代表整個問題 UNSAT；若所有 variables 都被指定完且沒有 conflict，代表找到 SAT model。中間那些學習、回跳、重啟，看起來很複雜，但本質上都是在加速這兩個終局判斷。

    這種觀察讓我比較能把 SAT solver 想成一套會自己修正搜尋策略的程序，而不是一棵純粹暴力的 DFS。也因為這樣，後面的 decision ordering、restart 和 BCP 才不再是零散技巧，而是這個主循環裡最重要的三個槓桿。

    Decision Ordering, Restart and Incremental SAT

    Decision ordering 這件事對我來說很像在問：如果我現在只能做一個猜測，哪個變數最值得先猜？因為 SAT solving 的成本不是只看猜了多少次，而是看這些猜測會不會快速引發 propagation，或者快速逼出 conflict。若我總是先猜到對整體結構最有影響的 variable，搜尋樹通常會淺很多。

    Restart 則讓我意識到，一個 solver 並不是在同一棵樹上死撐到底才叫聰明。若目前這條搜尋路徑一直在某個局部結構附近打轉，適時重啟其實是在保留 learned clauses 的前提下換一個入口重來。也就是說，restart 不是把進度清空，而是把「已學到的知識」保留下來，把「暫時不好的 trail」丟掉。這種做法比我原本直覺中那種硬走到底的暴力 DFS 成熟很多。

    Incremental SAT 又是另一個我很喜歡的觀念，因為它幾乎就是 verification workflow 的日常。真實問題很少是只解一次就結束，我常常會解一串彼此高度相關的 instance，例如同一個網路多加一條 constraint、同一個 BMC frame 再往後展一層、或同一個 design 換另一個 monitor。如果 solver 每次都從零開始，那很多前面學到的 clause 與活動度資訊都浪費掉了。

    我現在會把這三件事看成同一個方向的不同切面。Decision ordering 在決定我先看哪裡，restart 在決定我什麼時候該換路，incremental SAT 則在決定我哪些舊知識值得帶到下一題。這三個槓桿一起作用，才讓 modern SAT algorithm 不只是正確，而是真的實用。

    Boolean Constraint Propagation (BCP, aka. Logic Implication)

    如果說 decision 是 solver 主動做的事，那 BCP 就是 solver 對自己決策的即時回應。每當某些 literals 被指定之後，所有 clause 的狀態都會變動。有些 clause 已經滿足，有些還沒決定，但也有一些 clause 會被逼到只剩最後一個 literal 還能為真。這時那個 literal 就不再是自由選擇，而是必須被推導出來。

    我覺得 BCP 最厲害的地方，是它把「一個決策的後果」盡可能往前展開。沒有 BCP，我每做一次 decision 只得到一個 bit 的資訊；有了 BCP，我常常一口氣得到一長串 implication。也就是說，solver 真正探索的不是單點決策，而是決策帶出的 closure。這讓搜尋變得比較像在處理一整片受影響區域，而不是一格一格慢慢試。

    這同時也解釋了為什麼 BCP 幾乎總是 solver 裡最耗時間的一塊。因為每做一次決策、每學一條 clause、每次回跳後重新開始 propagation，solver 都在大量重複問同一個問題：在目前 trail 下，哪些 clause 變成 unit 了？哪些 clause 已經 conflict 了？所以很多 SAT solver 的實際性能，最後不是敗在理論上的 backtracking，而是敗在 BCP 資料結構夠不夠省。

    我很喜歡把 BCP 想成「局部一致性檢查器」，因為這個名字很貼切。它不直接告訴我整題 SAT 或 UNSAT，但會不斷把當前決策的局部後果推到極限。也正因為這個局部推理夠強，後面的 conflict analysis 才會有足夠多的資訊可以提煉出 learned clause。

    [LN] Experiencing (miniSat) SAT Solver

    DIMACS Format

    DIMACS 之所以重要，不是因為它語法多漂亮，而是因為它幾乎成了 SAT 世界的通用接口。一個 DIMACS CNF 檔通常由三部分組成：c 開頭的註解、p cnf <#vars> <#clauses> 的 header，以及每一條以 0 結尾的 clause。對 solver 來說，這就夠了。

    我很喜歡這種極簡格式，因為它把 SAT 前端和 SAT 後端切得非常乾淨。前端完全可以來自 graph coloring、bounded model checking、planning 或 equivalence checking；只要最後吐出 DIMACS，後端 miniSat 就能接。這種鬆耦合設計正是 SAT 生態系能發展得很快的原因之一。

    這次我實際透過 GV 內建的命令

    SATSolve DIMACS -File <file>

    去餵 DIMACS 檔案，整個感受非常直接。前端編碼、檔案格式、solver 呼叫三者之間的界線很清楚，也讓我更能感受到 SAT solver 作為 backend 的角色。

    Graph Coloring Problem

    這次我選的問題是 graph coloring，它本身就是一個經典的 NP complete 問題。給定一張圖 \(G=(V,E)\) 和顏色數 \(k\)，我要判斷能不能替每個 vertex 指派一個顏色，使得每條 edge 的兩端顏色都不同。這個問題我很喜歡，因為語意很直觀，而且一轉 SAT 就能很清楚地看到 variable、clause 和 constraint 之間的對應。

    我把變數定義成

    \[
x_{v,c} =
\begin{cases}
1, & \text{vertex } v \text{ 使用顏色 } c \\
0, & \text{否則}
\end{cases}
\]

    這樣一來，整個問題就變成在問：能不能找到一組 \(x_{v,c}\) 的 0/1 指派，同時滿足「每個點至少一種顏色、至多一種顏色、相鄰點不能同色」這三類限制。

    我這次刻意不用手工小到不行的圖，而是寫了一個固定 seed 的產生器，做四組 60 個 vertex、4 個 color 的 case。SAT case 用 planted coloring 生成，只在不同 planted color 的點之間隨機加邊；UNSAT case 則在同樣背景圖上額外塞進一個 5 點 clique，強迫 chromatic number 至少變成 5。這樣我同時保留了可重跑性、SAT/UNSAT 對照、以及 sparse / dense 對照。

    Encode Graph Coloring into CNF

    對 graph coloring 來說，CNF 編碼非常規則。若圖有 \(n\) 個 vertices、\(k\) 個顏色，那變數總數就是：

    \[
\#vars = nk
\]

    子句則來自三類限制。第一類是每個 vertex 至少要有一個顏色：

    \[
\bigvee_{c=1}^{k} x_{v,c}
\]

    第二類是每個 vertex 不能同時選兩個顏色：

    \[
\neg x_{v,c_1} \lor \neg x_{v,c_2},
\qquad c_1 \neq c_2
\]

    第三類是每條 edge \((u,v)\) 的兩端不能拿同一色：

    \[
\neg x_{u,c} \lor \neg x_{v,c}
\]

    因此若我用最直接的 pairwise 「at most one」編碼，總 clause 數可寫成：

    \[
\#clauses = n \left(1 + \binom{k}{2}\right) + |E|k
\]

    我這次故意不加 symmetry breaking，因為我想先看最原始的 miniSat 行為，不想一開始就把顏色對稱性用額外 constraints 壓掉。這樣做雖然可能讓 SAT case 比較容易出現多組等價解，但也比較能觀察 solver 在原始搜尋空間中的反應。

    Running miniSat through GV

    我這次不是直接呼叫外部 miniSat binary，而是走 GV 內嵌的 SATSolve DIMACS。這樣做的好處是，我整個流程都留在同一套工具內，也比較符合這週主題本來想讓我體驗的方向。實際命令很單純：

    python3 codex_tmp/graph_coloring_to_dimacs.py \
  --output codex_tmp/graph_coloring_hard/sat_60_4_p35.dimacs \
  --vertices 60 \
  --colors 4 \
  --prob 0.35 \
  --seed 4801 \
  --mode sat

cd gv
./gv <<'EOF'
SATSolve DIMACS -File /media/sf_SoCV_Learning_Notes/codex_tmp/graph_coloring_hard/sat_60_4_p35.dimacs
q -f
EOF

    不過我在真的整理數據時，發現目前 GV 這條 DIMACS 路徑有一個很小但很關鍵的問題。內建 MiniSat 會印一張 progress table，可是那一列數值是在真正 search() 之前先印出的，所以如果我直接抄那一列，conflicts 很容易看起來像 0。這會讓實驗表失真。

    因此我這次補了一個很小的修補，只改 gv/src/sat/minisatMgr.cpp 的 solve_dimacs_cnf() 輸出層，讓它在 solve() 結束後再額外印出：

    STATS conflicts=... decisions=... propagations=... clauses=... vars=...

    這樣我就能拿到真正的最終統計，而不是 search 前的快照。我很喜歡這種修補，因為它沒有碰 solver 核心演算法，只是把原本 solver 已經維護好的 stats 明確曝光出來，對這次的實驗剛好夠用。

    SAT / UNSAT, Sparse / Dense Experiments

    我最後固定做了四組 case，全部都是 60 個 vertex、4 個 color，差別只在密度和 satisfiability。這樣我比較不會把「問題大小差太多」和「SAT / UNSAT 差異」混在一起。

    Case Edges Clauses Result Conflicts Decisions Propagations Elapsed (s)
    sat_60_4_p35 462 2268 SAT 0 15 240 1.59
    sat_60_4_p65 865 3880 SAT 3 17 417 1.05
    unsat_60_4_p35 466 2284 UNSAT 66 105 4218 2.23
    unsat_60_4_p65 909 4056 UNSAT 67 78 4404 0.59

    我覺得這張表最好看的地方，是 SAT 和 UNSAT 的差異非常明顯。兩組 SAT case 幾乎沒什麼 conflict，solver 很快就找到一組合法著色；但一到 UNSAT，conflicts 和 propagations 都直接跳了一個數量級，代表 solver 真的得把大量局部可能性關掉之後，才能證明根本沒有解。

    另一個有趣的點是，dense_unsat 雖然 clause 比 sparse_unsat 多非常多，但 wall time 反而更短。這件事我一開始覺得有點反直覺，後來想通了，因為更密的圖同時也帶來更強的約束。換句話說，雖然 solver 要處理更多 clause，但 propagation 也更有力，很多錯誤分支能更早被剪掉，所以總時間不一定會跟 clause 數單調成長。

    What do the statistics tell me?

    這次我最大的體會是，MiniSat 的數據不能只看 SAT 或 UNSAT 這個二元結果。真正有意思的是搜索過程的形狀。像這四組 case 裡，conflicts 和 propagations 幾乎一眼就把「找到一組解」和「證明整個空間沒解」區分開來了。SAT case 只要有一條路夠快通到底就結束，UNSAT case 則必須把所有 relevant 分支都堵死。

    我也更能理解為什麼 BCP 在實務上這麼重要。這四組案例裡，真正差很多的不是 decisions，而是 propagations。例如 unsat_60_4_p35 和 unsat_60_4_p65 的決策次數分別是 105 和 78，但 propagation 都已經上看四千多次。也就是說，solver 真正花力氣的地方，是在每次 decision 之後把局部 implication 推到極限，而不是只做表面上的 branching。

    這組實驗也讓我更願意把 SAT solver 看成一個「encoding sensitive」的工具。問題同樣都是 graph coloring，可是只要我改變圖密度、是否保證 SAT、是否額外塞入 clique，solver 的 search shape 就完全不同。這和前一週我學到的 symbolic method 很像，真正決定工具好不好跑的，往往不是抽象問題名稱，而是我怎麼表示它。

    我這次保留的原始 GV 統計摘要
    sat_60_4_p35:  SAT,   conflicts=0,  decisions=15,  propagations=240,  elapsed=1.59s
sat_60_4_p65:  SAT,   conflicts=3,  decisions=17,  propagations=417,  elapsed=1.05s
unsat_60_4_p35: UNSAT, conflicts=66, decisions=105, propagations=4218, elapsed=2.23s
unsat_60_4_p65: UNSAT, conflicts=67, decisions=78,  propagations=4404, elapsed=0.59s
