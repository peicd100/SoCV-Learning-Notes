## 06. Introduction to Boolean Satisfiability

### Recap: BDD-Based Verification

我覺得這一週最重要的轉折，是我開始正面承認前面那條 BDD-based verification 路線雖然很漂亮，但它不是沒有代價。前幾週我一直在享受 BDD 的 canonical representation 和 shared subgraph 帶來的壓縮效果，可是一旦問題變成 sequential system 的 reachable states，我就很難再用「BDD 很會壓」這句話輕鬆帶過去。因為 reachable state set 的本質還是可能指數成長，而 BDD 只是讓我有機會把它壓小，不是保證一定壓得住。

如果把 safety property 寫成 `R ⟹ P`，那前面的做法其實很直覺，我先把 reachable states \(R\) 算出來，再去檢查這個集合是不是完全落在 property \(P\) 裡。問題在於，真正困難的常常不是 `P` 本身，而是 `R`。也就是說，我不是輸在 assertion 寫不出來，而是輸在「所有真的走得到的 state」太多，BDD 即使再會共享，還是可能被 register 數量拖到 memory explosion。

這裡我第一次很明確感受到，verification 的瓶頸有時候根本不在 specification，而在 state exploration。如果我一開始就把所有 reachable states 當成必經之路，那我其實是在逼自己先解一個更大的問題，再回頭證明 property。這種做法當然完整，可是也很可能太貴。

這一段讓我有個很重要的 mindset change。以前我會覺得 formal verification 的厲害之處在於「它可以看全部」。現在我比較願意把它理解成「它會先把我要回答的問題換成一個更可計算的形式」。如果前一種形式太大，就要找另一種形式，不然再漂亮的資料結構也撐不起來。

### The counter-example for \(R \Rightarrow P\) is \(R \land \neg P\)

當我把 `R ⟹ P` 換成反證法來看，整個問題突然清楚很多。要證明一個 property 成立，其實不一定要直接做「正面證明」；我只要證明它的 counterexample 不存在就好。對 `R ⟹ P` 來說，counterexample 正好就是：

\[
R \land \neg P
\]

這個式子對我來說很有力量，因為它把 verification 問題從「證明全部都對」改寫成「找不找得到一個會出錯的 assignment」。一旦問題被改寫成 existence question，我馬上就可以把它和 satisfiability 連起來看。如果 `R \land \neg P` 可滿足，代表真的存在一個 reachable bad state；如果它不可滿足，代表 reachable state set 根本碰不到 violation。

我很喜歡這個改寫，因為它讓「沒有反例」變成一個可運算的目標。從這裡往下走，不管我要用 BDD、SAT、BMC 還是其他 symbolic method，本質上都在問同一件事：是不是存在一個 assignment 或一條 trace 讓 violation 成立。也就是說，形式驗證真正強的地方不是只會說「對」或「錯」，而是它把 correctness 問題翻成一個可以被工具求解的存在性問題。

這也解釋了為什麼 SAT 會在這裡自然登場。因為當我已經把目標寫成「某個布林條件是否可滿足」，那下一步幾乎就是在問 Boolean satisfiability 了。這個轉換讓我覺得 SAT 不是突然插進來的新主題，而是前面 reachability 論述一路推下來的自然結果。

### Boolean Satisfiability (SAT) Problem

SAT 的形式定義其實比我原本想像得樸素很多。給定一個布林網路

\[
F : \mathbb{B}^n \mapsto \mathbb{B},
\qquad \mathbb{B} = \{0, 1\}
\]

我要做的事情只是找一組輸入 assignment \(A = \{x_1=a_1, x_2=a_2, \dots, x_n=a_n\}\)，讓 \(F(A)=1\)。如果找得到，這個 instance 就是 SAT；如果不管怎麼指定都不行，那它就是 UNSAT。

我覺得這個定義重要的地方，在於它非常中性。它沒有假設問題來自 verification、routing、planning、cryptography，甚至沒有假設我手上的東西一定是一個電路。只要我能把問題轉成一個布林條件，並且把「成功」寫成 \(F=1\)，SAT solver 就能接手。這種抽象層級很高的接口，正是它後來變成通用 backend 的原因之一。

另外我也比較能理解為什麼 SAT 一直被拿來當 complexity theory 的代表問題。因為它幾乎是最乾淨的「存在一組布林賦值使條件成立嗎」這個問題。很多看起來完全不同的實務任務，最後都能被擠進這個問句裡。也就是說，SAT 的厲害不只在 solver 本身，也在於它是一個非常通用的問題語言。

## [LN] How can SAT be so useful nowadays?

### Why does worst-case complexity not tell the whole story?

我以前看到「SAT 是 NP-complete」這句話時，直覺上都會把它翻成「那應該很難用吧」。但這一週我比較能分清楚 worst-case complexity 和 real-world instance 之間的距離了。NP-complete 說的是 worst-case 下不應期待一個對所有 instance 都很快的多項式時間演算法，可是現實中丟給 solver 的 instance 根本不是均勻隨機抽樣來的，它們通常有很強的結構、模組性、重複性和語意約束。

這件事在 verification 例子裡尤其明顯。很多布林約束不是憑空寫出來的，而是從 RTL、控制流程、規格限制或 time-frame unrolling 系統性展開出來的。這些來源天生就有 shared structure，例如重複的 cone、同型模組、規律的 counter、以及受限的 environment assumption。從 complexity 的角度看，這些 instance 仍然屬於 SAT；但從 solver 的角度看，它們並不是完全沒有形狀的布林雜訊。

我現在比較能接受一個看似矛盾、其實很重要的事實：一個問題在理論上很難，不代表它在工程上就沒價值。真正關鍵的是我要不要把實際 instance 的結構留下來，讓 solver 能利用它。如果表示法把所有結構都打爛了，那 worst-case 的樣子就會比較像現實；如果表示法把結構保留下來，實務體感就完全不同。

### Why is SAT such a good fit for verification and EDA problems?

我覺得 SAT 和 verification 特別合拍，最根本的原因是很多 EDA 問題本來就在問 existence question。像 combinational equivalence checking 在問「是否存在一組輸入讓兩個電路輸出不同」；bounded model checking 在問「是否存在一條長度 \(k\) 的 trace 讓 violation 發生」；assertion falsification 在問「是否存在 reachable bad state 或 witness」。這些句子只要再往下推一步，幾乎都會自然變成 SAT。

這種契合度很重要，因為它代表我不是硬把問題塞進 solver，而是問題本身就和 solver 的問句方向一致。相較之下，如果一個問題原本問的是最佳化、概率、數值穩定性或連續變量關係，那它轉 SAT 時通常會多很多折損；但 verification 的 counterexample search 本來就和 satisfiability 很接近，所以中間的語意損失比較小。

另一個我很有感的點，是 SAT 可以很好地扮演「最後一公里」的引擎。前端工具負責把電路、時間展開、property、環境限制整理成布林約束，後端 solver 只負責回答 SAT 或 UNSAT。這種前後端分離讓整個工具鏈很好擴張，因為我可以一直改前端建模、抽象、切分方式，而不必每次都重新發明一顆 solver。

### Why is CNF modeling a big deal instead of just a syntax choice?

剛開始學 SAT 時，我會把 CNF 想成一種有點無聊的標準格式，好像只是 solver 喜歡吃 clause，所以大家才把東西轉成 clause。現在我比較能理解，CNF 的價值不只是「統一輸入格式」，而是它把很多不同來源的限制都壓成一個可以穩定組合、穩定交換、穩定儲存的低階接口。

更重要的是，實務上常用的 Tseitin 風格轉換不需要把整個公式展開成指數大小。它允許我引入輔助變數，用近線性的大小把複雜布林網路轉成 CNF，同時保留很多原本的結構。這件事對我來說是 SAT 真正可用的基礎，因為如果每次從電路轉 CNF 都先爆炸，那 solver 再強也沒有意義。

我覺得 CNF 像是一種工程上的匯流排。所有前端問題最後只要都能整理成 literal 和 clause 的世界，後端 solver 就能重用；而 solver 的進步也會立刻回饋到所有建模前端。這種可組合性比表面上看起來重要得多，它讓 SAT 不只是單一問題，而是一個可重用的平台接口。

### Why did a solver ecosystem matter as much as the core algorithm?

如果只講「SAT 有厲害演算法」，我覺得故事其實只說了一半。另一半是 solver ecosystem 真的成熟起來了。從 Chaff 把工程實作細節做到極致，到 MiniSat 刻意把 solver 寫得小而清楚、方便擴充，我看到的不是只有理論突破，而是一整套讓人願意把 SAT 接進自己系統裡的工程文化。

這件事的影響非常大。因為一旦有一個夠小、夠快、夠穩、又容易嵌入的 solver，很多領域的人就不用從零造輪子，只要把自己的問題轉成 CNF 或 assumptions 就能開始實驗。也就是說，solver 不只是回答問題，它還降低了整個社群使用 SAT 的門檻。

我覺得這也是 SAT 特別像「平台技術」而不只是「演算法成果」的原因。當 MiniSat 這類 solver 提供增量介面、可嵌入 source、清楚的 internal structure 時，SAT 就不再只是 paper 上的一個 result，而是很多人可以直接拿來做 domain-specific extension 的基礎元件。這種可擴展性和可重用性，本身就是它變得實用的重要原因。

## SAT Problem Formulation in CNF

### SAT Problem Formulation in CNF

CNF 的形式我現在會用最基本的 clause language 來理解。整個公式是很多 clause 的 conjunction，而每個 clause 又是一些 literal 的 disjunction。例如

\[
(a+b+c)(a'+b'+c)(a'+b+c')(a+b'+c')
\]

在這裡，`a`、`b`、`c` 是 variables，每一對括號是一個 clause，而 variable 的正相或反相都是 literal。要讓整個 CNF 成立，條件不是「大部分 clause 成立」，而是每一個 clause 都必須至少有一個 literal 為 1。

這個表示法之所以對 solver 友善，是因為它把 constraint 的局部性保留得很好。每個 clause 都像是一條小規則，我只要有某些變數被決定，就能很快判斷這條規則是已滿足、尚未決定、還是已衝突。後面 BCP、unit propagation、learned clause 其實都建在這個局部觀察能力上。

我很喜歡這裡的「conflict」觀念，因為它非常工程化。如果某個 clause 裡所有 literal 都變成 0，那就不是抽象上的「可能有問題」，而是現在這組決策已經不可能成功。也就是說，CNF 不只在描述解，也在描述搜尋過程中的失敗訊號。

### Why can’t we model the SAT problem into a “Disjunctive Normal Form” (DNF, i.e. Sum-of-Product (SoP) form)?

這個問題一開始讓我很困惑，因為 DNF 看起來也很合理，為什麼 solver 不直接吃 DNF 就好。後來我想通了，SAT on DNF 之所以幾乎是 trivial，是因為只要有任何一個 product term 內部沒有自相矛盾，它就已經是一個 satisfying witness 了。換句話說，DNF 的 satisfiability 幾乎等於在找「有沒有一個 term 本身可滿足」，這和 CNF 完全不是同一個難度層級。

這也代表，如果我先把任意公式轉成 DNF，再去解 SAT，看起來像是把難題變簡單了，但真正的代價被藏在轉換過程裡。因為從一般布林公式或 CNF 轉成等價 DNF，通常會發生巨大的展開，大小可能直接指數爆炸。也就是說，DNF 的「簡單」不是免費的，它只是把困難前移到表示法展開。

我覺得這個例子很有教育意義，因為它提醒我不要把 normal form 當成純格式問題。格式本身就決定了哪一類運算是便宜的、哪一類會爆炸。對 SAT 來說，CNF 之所以成為主流，不是因為它漂亮，而是因為它和搜尋、衝突、propagation 的交互作用非常自然。

### A Brief History of SAT Algorithms

我現在回頭看 SAT 的演進，會覺得它其實就是一段「從數學定義走到工程可用」的歷史。最早的問題是 `2^n` 級別的暴力決策，看起來幾乎沒有實務可能；接著大家開始發展 DPLL 這種系統性的 branch-and-prune；再往後才慢慢把 propagation、learning、restart、資料結構與 heuristic 全部疊起來。

這段歷史對我最有啟發的地方，是 solver 的進步不是單點突破，而是一層一層累積起來的。很多看似不起眼的細節，例如 clause 資料結構、如何記錄衝突、什麼時候 backtrack、如何重用已學到的資訊，最後都會變成數量級差異。SAT 變得實用，不是因為突然有一個魔法公式，而是因為整條搜尋鏈上的摩擦都被一點一點降下來了。

這也讓我比較能理解，為什麼同樣都叫 SAT solver，不同世代之間的體感差異會這麼大。形式上大家都在解 satisfiability，可是真正決定實力的，是它們怎麼處理 search space，而不是只看它們有沒有在做 search。

### A Very Basic SAT Algorithm

最基本的 SAT algorithm 在概念上其實很直白。我先挑一個尚未指定的 variable，試著給它一個值；如果目前所有條件都還沒衝突，就繼續往下做；如果走到底所有 variable 都指定完，代表找到 model；如果中途衝突，就 backtrack，改試另一個值。這種做法本質上就是系統化的深度優先搜尋。

我覺得這個版本很適合當起點，因為它把 SAT solving 的核心動作全都露出來了，像是 decision、propagation、conflict、backtrack。即使後面的 CDCL 看起來複雜很多，它也不是完全不同的東西，而是在這條基本骨架上加了更聰明的記憶和跳躍機制。

但這個基本版的限制也很明顯，因為它幾乎每次撞牆都只得到一句「這條路不行」，卻沒有真的學到新知識。如果我只是 chronologically 回上一層，很多其實已經知道必敗的局部組合，後面還是可能反覆踩到。這也是為什麼後面一談到 CDCL，我馬上就能理解它的價值：它不是把 search 拿掉，而是讓 search 不要白走。

### Modern SAT Improvement #1: Conflict-Driven Clause Learning (CDCL), aka Conflict-Driven Non-Chronological Backtracking

我會把 CDCL 理解成一個很工程、也很聰明的升級版本。最基本的想法是，當我撞到 conflict 時，不要只知道「這條路錯了」，而是要把這次衝突整理成一條新的 clause，讓 solver 之後永遠記得這種組合不值得再試。這樣 learned clause 就像 search process 自己累積出來的經驗。

另一個更有感的地方，是 non-chronological backtracking。基本版演算法通常只退一層，但 CDCL 會問：「真正造成衝突的關鍵決策是哪些？」如果我能把衝突原因抽成 learned clause，那就沒必要一層一層慢慢退，而是可以直接跳回真正有意義的 decision level。這種 backjumping 對我來說非常像 debug 時直接找到 root cause，而不是從最後一層 call stack 慢慢往回翻。

雖然這一週的範圍只走到 CDCL 的入口，還沒進 implication graph、UIP 和 learning 細節，但我已經很能感受到它為什麼會成為現代 SAT 的分水嶺。因為從這一刻開始，solver 不再只是被動地搜尋，而是會把衝突轉成知識，把知識再丟回搜尋本身。這種「search with memory」的味道，才是 SAT 從理論題目變成實戰工具的關鍵。
